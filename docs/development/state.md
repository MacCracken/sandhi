_(1.6.12 — 2026-06-23 — **Default mDNS resolver now actually receives (QU two-socket fix).** The default unicast-response (QU) `.local` resolver `sock_connect`ed to the mDNS group then `sock_recv`ed on the *same* socket — so Linux's connect()-source-filter dropped every reply (the answer comes from the responder's own unicast IP, not the group: the 1.5.4 bug, never fixed on the QU path). A second latent bug compounded it: it used a random TXID, but mDNS responses carry ID=0 (RFC 6762 §18.1), so even a received reply would be rejected on ID mismatch. The 1.5.5 QM fix didn't transfer trivially — a QU reply is unicast to the *querier's source port*, not the group. **Fix** (`src/discovery/local.cyr` only, pin stays 6.2.37): factored a shared `_sandhi_local_query_2sock_a(a, name, qclass_hi, tx_port)` (module reordered so the RX/recv helpers precede the default resolver). RX is **unconnected**, bound 5353 + `SO_REUSEPORT` + group-joined (sees the answer from any source); TX is connected to the group, send-only. For QU the TX binds **5353** so the unicast reply lands on RX's port (the connect()-filter on the TX rejects it — source ≠ group — so it falls to the unconnected RX); the group-join also catches a multicast answer; ID=0 now. The opt-in QM resolver (1.5.5) now shares the helper (`0x00, 0` — ephemeral TX; byte-identical, its loopback gate still green). `sandhi_discovery_local_available()` is now AGNOS-aware (0 there — the default joins the group now; no verb signature change). **Verified**: **1112 assertions green** (sandhi 540 / h2 167 / alloc 342 / rpc 63; sandhi 539 → 540: the new `discovery/local/qu_unicast` loopback gate — an unconnected RX receives a **unicast** datagram even with a connected TX coexisting on the same port via `SO_REUSEPORT`, the dispatch crux a single connect()ed socket would fail); QM gate + discovery smoke unchanged; smoke + `CYRIUS_DCE=1` OK; `cyrius lint src/discovery/local.cyr` 0/0; `dist/sandhi.cyr` at v1.6.12. **Live-LAN note**: validated via the loopback dispatch gate (the 1.5.5 standard) + the structural argument — no mDNS responder was reachable here (avahi not running).
